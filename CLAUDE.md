# KWANUS — Locked Product Doctrine

## READ BEFORE ANY WORK
Start every session by reading: `docs/foundation/START_HERE.md`

## What Is KWANUS
KWANUS is a **Financial Outcome Engine**. Not a credit repair app. Not a fintech tool. Not a feature stack.

## Core Loop
```
Repair -> Build -> Optimize -> Prepare -> Fund
```
Every feature must support this loop directly. If it doesn't, defer it.

## Control Documents
All future work must follow these canonical files:
- `docs/SYSTEM_STATE.md` — current phase and status
- `docs/ROUTE_MAP.md` — all application routes
- `docs/DISPUTE_STRATEGY_SPEC.md` — dispute engine brain
- `docs/PROFILE_BUILD_SPEC.md` — profile build engine brain
- `docs/FUNDING_GUIDANCE_SPEC.md` — funding guidance brain
- `docs/PHASE_HISTORY.md` — phase completion record
- `docs/foundation/` — full doctrine, roadmap, architecture, compliance

## Decision Filter
Before building anything, ask:
> Does this improve clarity, trust, compliance, readiness, conversion, retention, or user outcomes?

If NO → defer. If UNCLEAR → ask first.

## Compliance (Non-Negotiable)
- No guarantee language
- No "we will fix your credit"
- No acting on behalf of users
- Educational and consultant posture only
- See `docs/foundation/COMPLIANCE_GUARDRAILS.md`

## Build Rules
1. Map every task to the roadmap
2. Follow the build protocol
3. Protect compliance posture
4. Strengthen the core loop
5. Do not add disconnected features
