# HIMALAYA — UX Doctrine

## The Biggest Rule

> Himalaya must always reduce friction, not create more of it.

## Core Feeling

The user should feel:
- "This system understands where I am"
- "It told me what to do without making me feel dumb"
- "I got structure and assets without doing hours of work"
- "It feels like it was built for my situation"

## Design Principles

### 1. Minimal Questions, Maximum Intelligence
- Ask the fewest possible questions to get started
- Infer what you can from context (URL scan, business type, niche)
- Smart defaults everywhere
- Every question must earn its place

### 2. Adaptive Flow
- The system changes based on the user's answers
- Beginners get more guidance, more explanation, fewer choices
- Operators get faster paths, more control, less hand-holding
- Never show everything at once

### 3. One Purpose Per Screen
- Every screen answers one question or enables one action
- No competing information
- No multi-purpose dashboards that overwhelm
- Secondary info is accessible but not fighting for attention

### 4. Show the Why
- Every recommendation includes reasoning
- Not too much — just enough to build trust
- "We suggest this because..." not just "Do this"
- Makes the system feel intelligent, not random

### 5. Progressive Disclosure
- Start simple, reveal complexity as needed
- Don't front-load features
- Advanced options exist but don't clutter the default experience
- The user peels layers, not digs through piles

### 6. Never Make the User Feel Dumb
- No jargon without explanation
- No assumptions about business knowledge
- Error messages are helpful, not technical
- Empty states guide, not scold
- Loading states inform, not just spin

### 7. Connected, Not Disconnected
- Front-end connects to back-end (site → CRM → automation → follow-up)
- A business is not just a page — it's a system
- Every generated asset is part of a larger whole
- The user should see how pieces connect

## Visual Standards

- Dark, premium, clean
- Generous whitespace
- Card-based layouts
- Subtle status colors (green/amber/red)
- Smooth transitions
- System feels expensive, not DIY

## Interaction Standards

- Max 2-3 clicks to any primary action
- Forms: one step at a time, not one giant form
- Selections: cards over dropdowns when possible
- Actions: clear verb + object ("Generate Site", "Review Plan", "Fix This")
- Feedback: every action produces visible feedback

## Beginner vs Operator Mode

| Element | Beginner | Operator |
|---------|----------|----------|
| Guidance | Step-by-step, contextual | Minimal, on-demand |
| Defaults | Pre-selected, explained | Editable, customizable |
| Layout | Focused, one thing at a time | Dense, multi-panel |
| Language | Plain, supportive | Direct, technical OK |
| Speed | Guided pace | Self-paced, fast |

## Anti-Patterns (Never Do This)

- Giant forms with 15+ fields
- Dashboards with 10+ competing widgets
- Features with no explanation of purpose
- Raw data without interpretation
- Empty screens with no guidance
- Modals stacked on modals
- Forcing strategy language on beginners
- Making the user search for the next step
