# KWANUS — Agent Operating Instructions

## Identity Command

You are now building KWANUS under a **locked product doctrine**.

KWANUS is a financial outcome engine, not a random feature stack, not a generic credit repair app, and not a hype-based fintech tool. Its job is to reduce chaos and guide users through the strongest sequence of financial actions.

## Core Loop

The product must stay focused on one core outcome loop:

```
Repair -> Build -> Optimize -> Prepare -> Fund
```

Every feature must support that loop directly.

## Decision Filter

Before building any feature, ask:

> Does this improve clarity, trust, compliance, readiness, conversion, retention, or user outcomes?

If **no** → defer it.
If **unclear** → ask before building.

## Build Rules

1. Read `docs/foundation/START_HERE.md` before any session
2. Every build task must map to the `PRODUCT_ROADMAP.md`
3. Follow the `BUILD_PROTOCOL.md` for every change
4. Protect the compliance posture defined in `COMPLIANCE_GUARDRAILS.md`
5. Match the tone defined in `COPY_DOCTRINE.md`
6. Match the interface feel defined in `UI_DOCTRINE.md`
7. Do not add features not on the roadmap
8. Do not refactor working systems without explicit approval
9. Do not introduce dependencies on unbuilt phases

## UX Rules

1. The system does the thinking. The user sees only what they need.
2. Every screen has one clear purpose and one clear next action.
3. Language is educational, consultant-style, compliance-safe.
4. No guarantee language. No legal representation language. No "we will fix your credit" language.
5. The experience must feel calm, premium, and structured.

## Communication Rules

1. When uncertain, ask — do not assume.
2. When a task is ambiguous, request clarification tied to the roadmap.
3. When something conflicts with the doctrine, flag it before building.
4. When proposing a feature, explain how it maps to the core loop.

## Forbidden Actions

- Adding features not on the roadmap without approval
- Using guarantee or promise language in any UI copy
- Implying KWANUS acts on behalf of users (it educates and guides)
- Skipping phases
- Building Phase N+1 features during Phase N
- Removing or weakening compliance guardrails
