# HIMALAYA — Agent Operating Directive

You are building Himalaya under a **locked product doctrine**.

## Identity
Himalaya is a guided business operating system that helps people start, structure, optimize, and scale businesses. It is not a stack of tools. It is not a random AI wrapper. It is a guided transformation engine.

## Mandatory Pre-Work
Before any build session, read:
1. `docs/himalaya/START_HERE.md`
2. The relevant spec file for your task
3. `docs/himalaya/EXISTING_ASSET_MAP.md` (to avoid rebuilding what exists)

## Sacred Sequence (Never Break This)
```
Entry → Diagnosis → Strategy → Generation → Optimization
```
Generation never fires without diagnosis and strategy first.

## Who It Serves
- People with no business (start from scratch)
- People with a business but bad systems (diagnose and fix)
- People who need help choosing the right path (guide first)

## Control Files
- `docs/himalaya/PHASE_1_SPEC.md` — What Phase 1 delivers
- `docs/himalaya/CORE_ENGINES.md` — Engine definitions
- `docs/himalaya/USER_FLOWS.md` — User journey maps
- `docs/himalaya/PAGE_ARCHITECTURE.md` — Page structure
- `docs/himalaya/BUILD_PROTOCOL.md` — Build process
- `docs/himalaya/UX_DOCTRINE.md` — Experience standards
- `docs/himalaya/PRODUCT_ROADMAP.md` — What and when

## Rules
1. Every build task must map to the roadmap
2. Follow the sacred sequence for all features
3. Integrate existing systems — don't rebuild what works
4. Diagnosis before generation, always
5. Every screen has one purpose and one next action
6. Minimal questions, smart defaults, adaptive flow
7. Never make the user feel dumb
8. Beginner mode guides, operator mode moves fast
9. Do not add disconnected features
10. Do not build Phase N+1 during Phase N
