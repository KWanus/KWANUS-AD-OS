# KWANUS Agent Operating Directive

You are building KWANUS under a **locked product doctrine**.

## Identity
KWANUS is a financial outcome engine that creates order before money moves. It is not a random feature stack, not a generic credit repair app, and not a hype-based fintech tool.

## Mandatory Pre-Work
Before any build session, read:
1. `docs/foundation/START_HERE.md`
2. `docs/SYSTEM_STATE.md`
3. The relevant SPEC file for your task

## Core Loop (Sacred)
```
Repair -> Build -> Optimize -> Prepare -> Fund
```
Every feature must map to this loop. No exceptions.

## Control Files
- `docs/SYSTEM_STATE.md` — know what phase you're in
- `docs/ROUTE_MAP.md` — know what routes exist and why
- `docs/DISPUTE_STRATEGY_SPEC.md` — dispute engine logic
- `docs/PROFILE_BUILD_SPEC.md` — profile build logic
- `docs/FUNDING_GUIDANCE_SPEC.md` — funding guidance logic
- `docs/foundation/PRODUCT_ROADMAP.md` — what gets built and when
- `docs/foundation/BUILD_PROTOCOL.md` — how to build safely
- `docs/foundation/COMPLIANCE_GUARDRAILS.md` — what you cannot say or do

## Rules
1. Every build task must map to the roadmap
2. Follow the build protocol for every change
3. Protect compliance posture at all times
4. Strengthen the end-to-end user outcome loop
5. Do not add disconnected features
6. Do not skip phases
7. Do not build Phase N+1 features during Phase N
8. If a feature does not improve clarity, trust, compliance, readiness, conversion, retention, or user outcomes — defer it
9. The UX must remain calm, premium, friction-light, and structured
10. The system does the thinking — present only what the user needs at the right moment
